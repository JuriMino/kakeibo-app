Laravel Breeze導入後のセットアップ注意点

作成日: 2026-05-05 / プロジェクト: kakeibo-app (Laravel + Sail + Vite)

このドキュメントの目的

Laravelプロジェクトの作成までは問題なくできたが、 認証機能のためにLaravel Breezeをインストールしてから複数回エラーが発生した。 次回以降、同じハマりどころで詰まらないようにするためのメモ。

環境

項目内容
OSmacOS (Darwin 24.6.0)
開発環境Laravel Sail (Docker)
フロントビルドVite 8.x
パッケージマネージャnpm
コンテナ内パス/var/www/html

1. Breezeインストールで起きたエラー

エラー① axiosが見つからない

[plugin:vite:import-analysis] Failed to resolve import "axios"
from "resources/js/bootstrap.js". Does the file exist?
原因: resources/js/bootstrap.js は import axios from 'axios' をしているが、 package.json の dependencies に axios が記載されていない。 Laravel 11以降のスケルトンでは axios がデフォルトで含まれていない構成があるため、 Breeze導入後にViteを起動すると import 解決に失敗する。

解決方法

# Sail（Docker内）で axios をインストール
./vendor/bin/sail npm install axios

# またはホスト側で
npm install axios

エラー② Cannot find native binding

Error: Cannot find native binding. npm has a bug related to optional dependencies
(https://github.com/npm/cli/issues/4828).
Please try `npm i` again after removing both package-lock.json and node_modules directory.
原因: ホスト（macOS）で npm install した場合、Mac用のネイティブバイナリ（rollup等）が入る。 しかし実際の npm run dev は Docker（Linux）内で実行されるため、 バイナリのプラットフォームが合わずに読み込み失敗する。

解決方法

# 1. ホストで作られた node_modules / package-lock.json を削除
rm -rf node_modules package-lock.json

# 2. Sail（Linuxコンテナ）内で再インストール
./vendor/bin/sail npm install

# 3. 開発サーバ起動
./vendor/bin/sail npm run dev
結果: localhost がエラーなしで表示されるようになった。

2. 次回以降のチェックリスト

Breezeインストール直後にやること

  1. php artisan breeze:install 実行後、 package.json に axios が含まれているか確認する。 含まれていなければ追加する。
  2. npm install は 必ず Sail（Docker）内で実行する。 
    ./vendor/bin/sail npm install
  3. npm run dev も 必ず Sail 内で実行する。 
    ./vendor/bin/sail npm run dev
  4. マイグレーションを忘れずに。 
    ./vendor/bin/sail artisan migrate

やってはいけないこと

  • ホスト側の npm install と Sail 側の npm install を混ぜて使う（バイナリ不整合の原因）。
  • 存在しないスクリプト（例: sail npm run clean-install）を実行する。package.json の scripts に定義があるものしか動かない。
  • エラーが出たときに node_modules だけ消して package-lock.json を残す（古いロックファイルが原因の不整合が再現する）。両方消すこと。

3. トラブル時の復旧手順（保存版）

# フロントが壊れたら、このセットで大体直る
rm -rf node_modules package-lock.json
./vendor/bin/sail npm install
./vendor/bin/sail npm run dev

それでも直らない場合:

# Sailコンテナを再ビルド
./vendor/bin/sail down
./vendor/bin/sail up -d --build

4. 用語メモ

用語意味
axios JavaScript用のHTTPクライアントライブラリ。Laravel + フロント間でAjax通信するための標準ツール。window.axios としてグローバルに使えるよう Laravel が bootstrap.js で読み込んでいる。
Vite フロントエンドのビルドツール。npm run dev で開発サーバが起動し、CSS/JSの変更が即時反映される。
Laravel Sail LaravelをDockerで動かすための公式ラッパー。./vendor/bin/sail 経由でコンテナ内コマンドを実行する。
Breeze Laravelの認証スターターキット。ログイン/登録/パスワードリセット等の画面とロジックを一括導入できる。

5. データベース名の不一致によるアクセス拒否エラー

発生したエラー

$ ./vendor/bin/sail mysql kakeibo_app -e "DESCRIBE expenses;"
mysql> use kakeibo_app;
ERROR 1044 (42000): Access denied for user 'sail'@'%' to database 'kakeibo_app'
原因: .env の DB_DATABASE が初期値の laravel のままだったため、 Sail起動時に自動作成されたのは laravel データベースで、 kakeibo_app は存在しなかった。 存在しないDBにアクセスしようとしたため「Access denied」と表示された （MySQLは「DBが無い」場合でも権限エラーとして返すことがある）。

確認手順

# .env のDB設定を確認
grep DB_ .env

# 実際に存在するDB一覧を確認
./vendor/bin/sail mysql -e "SHOW DATABASES;"

解決方法（プロジェクト名に合わせて kakeibo_app に統一する場合）

  1. .env を編集 
    DB_DATABASE=kakeibo_app
  2. Sailを再起動して compose.yaml の環境変数を読み直す （Sailは MYSQL_DATABASE: ${DB_DATABASE} で起動時にDBを自動作成する） 
    ./vendor/bin/sail down
./vendor/bin/sail up -d
  3. マイグレーションを実行 
    ./vendor/bin/sail artisan migrate
  4. 確認 
    ./vendor/bin/sail mysql kakeibo_app -e "DESCRIBE expenses;"
ポイント: プロジェクト作成直後（マイグレーションの前）に .env の DB_DATABASE をプロジェクト名と揃えておくのが一番ラク。 後から変えると、古いDBに作られたテーブルとの不一致が起きやすい。

やってはいけないこと

  • .env を変えただけで sail up -d し直さない。 compose.yaml は起動時に環境変数を読むので、変更後は 再起動が必須。
  • アクセス拒否エラーを見て、いきなり sail ユーザーの権限を疑う。 まず SHOW DATABASES でDBの存在を確認するのが先。

6. 学び

  • Docker（Sail）を使うときは、npm系のコマンドは原則すべてコンテナ内で実行すると覚えておく。
  • 「Cannot find native binding」が出たら、ほぼ ホスト/コンテナのバイナリ不整合を疑う。
  • エラーメッセージは英語でも丁寧に読むと、解決手順がそのまま書いてあることが多い（今回も remove both package-lock.json and node_modules と明記されていた）。
  • プロジェクト作成直後に .env の DB_DATABASE をプロジェクト名と揃えておくと、後でDB名不一致のハマりが起きない。
