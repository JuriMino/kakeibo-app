共通フォーム部品 (_form.blade.php) の value 記述 解説

作成日: 2026-05-06 / 対象: kakeibo-app の resources/views/expenses/_form.blade.php

このドキュメントの目的

新規作成画面と編集画面で同じフォームを使い回すために、 _form.blade.php を共通部品にしている。
その中で value 属性に複雑な式が出てくるが、 なぜそう書くのか・各演算子の意味が直感的に分かりにくいので、ここで一つずつ解説する。

共通フォームが満たすべき要件

同じフォーム部品でも、画面の状況によって value に入れたい値が違う。

状況valueに入れたい値
① 新規作成画面（最初の表示）空
② 編集画面（最初の表示）DBから取ってきた既存の値
③ バリデーション失敗で戻ってきた時ユーザーが入力した直前の値（消えると困る）

これら3つを1行で実現するのが old() + ヌル合体 + ヌル安全演算子の組み合わせ。

実際のコード（kakeibo-app から抜粋）

<!-- タイトル -->
<input type="text" name="title"
       value="{{ old('title', $expense?->title) }}">

<!-- 金額 -->
<input type="number" name="amount"
       value="{{ old('amount', $expense?->amount) }}">

<!-- 支出日 -->
<input type="date" name="spent_on"
       value="{{ old('spent_on', $expense?->spent_on?->format('Y-m-d')) ?? now()->format('Y-m-d') }}">

<!-- カテゴリ（select の selected 判定） -->
<option value="{{ $category }}"
        {{ old('category', $expense?->category) === $category ? 'selected' : '' }}>
    {{ $category }}
</option>

<!-- メモ -->
<textarea name="memo">{{ old('memo', $expense?->memo) }}</textarea>

1. {{ ... }} （Blade出力）

Blade テンプレートで {{ $value }} と書くと、 PHPの値をHTMLとして出力する。
自動でHTMLエスケープされるので、ユーザー入力をそのまま埋め込んでもXSS対策になる。 

{{ $expense->title }}    // <script> などが入っていても安全に表示される

2. old() ヘルパー

Laravelが提供する関数。 「直前のリクエストでユーザーが送信した値」をセッションから取り出す。 バリデーション失敗で画面が戻ってきた時に、入力欄を空にしないために使う。

シグネチャ

old(string $key, mixed $default = null)
引数意味
第1引数 'title'セッションから探したい入力名（input の name属性）
第2引数 $expense?->titleold が無かった時のフォールバック値

動きの違い

状況old('title', $expense?->title) が返す値
新規作成（初回表示）old なし、$expense は null → null（空）
編集（初回表示）old なし → $expense->title
バリデーション失敗で戻ってきたold あり → ユーザーが直前に入力した値
ポイント: 「優先順位」で覚える。
① ユーザーが直前に入れた値（old） → ② DBにある既存値（$expense） → ③ 空。

3. ?-> ヌル安全演算子（PHP 8.0+）

オブジェクトが null の時にプロパティアクセスしてもエラーにならない演算子。 

$expense?->title

// 等価なコード:
$expense !== null ? $expense->title : null

なぜ必要か

create.blade.php から _form.blade.php を呼ぶ時、 まだDBに保存されていないので $expense = null を渡している。 

<!-- create.blade.php -->
@include('expenses._form', [
    'expense' => null,        ← null を渡す
    'categories' => $categories,
    'submit' => '登録する',
])

もし普通の $expense->title と書くと、新規作成時に 「nullに対してプロパティアクセスした」エラーになる。 ?-> を使えば null の時はそのまま null を返してくれる。

NG: $expense->title → 新規作成画面でエラー
OK: $expense?->title → 新規作成画面では null になるだけ

連鎖もできる

$expense?->spent_on?->format('Y-m-d')

// 意味: $expense が null なら null。
//      spent_on が null でも null。
//      両方あれば 'Y-m-d' でフォーマット。

4. ?? ヌル合体演算子

「左辺が null なら右辺を使う」演算子（PHP 7.0+）。 

$x ?? 'デフォルト値'

// 等価なコード:
isset($x) ? $x : 'デフォルト値'

支出日の例で見る

{{ old('spent_on', $expense?->spent_on?->format('Y-m-d')) ?? now()->format('Y-m-d') }}

分解すると次の優先順位:

  1. old('spent_on', ...) → バリデーション失敗時のユーザー入力値
  2. $expense?->spent_on?->format('Y-m-d') → 編集時のDB値
  3. ?? now()->format('Y-m-d') → どちらも null なら今日の日付を入れる
支出日だけ ?? を使っているのは、新規作成画面で 「最初から今日の日付が入っていると入力が楽」だから。 タイトルや金額は空欄スタートが自然なので不要。

5. 三項演算子 ... ? 'A' : 'B' （selected判定）

{{ old('category', $expense?->category) === $category ? 'selected' : '' }}

意味:

  • 左辺：「現在選ばれているべきカテゴリ」（old or DB値）
  • 右辺：「ループ中の選択肢」（$category）
  • 一致したら selected を出力、違えば空文字

これにより、編集画面を開いた時に保存済みカテゴリが最初から選ばれた状態になる。

6. 全部つなげて読む練習

例: 支出日の value

value="{{ old('spent_on', $expense?->spent_on?->format('Y-m-d')) ?? now()->format('Y-m-d') }}"

新規作成画面で読むと:

  1. old('spent_on', ...) → 初回なので old は無い。第2引数を見る。
  2. $expense は null なので、$expense?->spent_on は null。さらに ?->format も null。
  3. old全体としては null が返る。
  4. ?? の右辺 now()->format('Y-m-d') が使われる。
  5. 結果: 今日の日付が入った状態でフォームが開く。

編集画面で読むと:

  1. old('spent_on', ...) → 初回なので old は無い。第2引数を見る。
  2. $expense は支出オブジェクト。spent_on は Carbon インスタンス。
  3. $expense?->spent_on?->format('Y-m-d') → '2026-05-06' のような文字列が返る。
  4. これは null ではないので ?? の右辺は使われない。
  5. 結果: DBに保存されている日付がフォームに入る。

バリデーション失敗で戻ってきた時:

  1. old('spent_on', ...) → 直前にユーザーが入力した値が返る。
  2. その時点でフォールバックは見ない。
  3. 結果: 入力した値が消えずに残る。

7. まとめ表

8. よくあるハマりどころ

  • create.blade.php で expense を渡し忘れる → 「Undefined variable $expense」エラー。null でも良いので必ず渡す。
  • ?-> を使わずに -> を使う → 新規作成画面で「null上のプロパティアクセス」エラー。
  • old('title', $expense->title) と書く → 上と同じ。$expense?->title にする。
  • spent_on を文字列のまま比較 → date型inputには Y-m-d 形式の文字列が必要。Carbonインスタンスのままだとブラウザが認識しないので format() で文字列化する。

9. 参考: 共通部品の呼び出し方

<!-- create.blade.php （新規作成） -->
@include('expenses._form', [
    'expense'    => null,
    'categories' => $categories,
    'submit'     => '登録する',
])
<!-- edit.blade.php （編集） -->
@include('expenses._form', [
    'expense'    => $expense,
    'categories' => $categories,
    'submit'     => '更新する',
])

この2つで $expense の中身だけ変えれば、同じフォーム部品が 両方の画面で正しく動くようになる。それを支えているのが old() と ?-> と ?? の組み合わせ、というわけ。
